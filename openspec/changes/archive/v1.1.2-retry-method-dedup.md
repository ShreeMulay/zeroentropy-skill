# Change: v1.1.2 Retry Hardening, Method-Order & Dedup

## Status
completed

## Description
Address three review findings from the full codebase review: narrow retry semantics to genuine transient errors (M2), fix the SDK method-probe order to match the real camelCase API (M1), and extract a shared `buildContent` helper to remove index/batch duplication (L1).

## Motivation
- **M2 (bug, P1):** `withRetry` retries on *any* error lacking an HTTP status (`index.ts:59-64`). Programmer errors (TypeError, malformed-response crashes) and non-transient failures (auth/DNS misconfig) are retried 5× with backoff (~15s wasted) and the real error is obscured.
- **M1 (bug, P2):** `list_collections` probes `get_list` (non-existent) before `getList` (the real SDK method). Inverted order is misleading and forces `as any` casts.
- **L1 (task, P2):** `zeroentropy_index` and `zeroentropy_batch` contain identical content-switch blocks; divergence risk + ~30 redundant lines.

## Impact
- `plugin/src/index.ts` — `isRetryableError` narrowing; reorder method probes (getList/getInfo first); add `buildContent()` helper used by index + batch
- `plugin/tests/index.test.ts` — tests for: non-retryable TypeError fails fast (1 attempt); retryable ECONNRESET retries; getList called directly; buildContent parity for text/text-pages/auto in both tools
- `CHANGELOG.md` — v1.1.2 entry

## Acceptance Criteria
- [x] A no-status `TypeError` (programmer error) is NOT retried (attempts == 1)
- [x] A no-status network error (ECONNRESET/ETIMEDOUT/etc.) IS retried (attempts == MAX_RETRIES+1)
- [x] 429/5xx still retried; 4xx still fail fast (unchanged)
- [x] `list_collections` invokes `getList` directly (test asserts it)
- [x] `buildContent` shared by index + batch; both produce identical content objects for text / text-pages (with pages) / auto
- [x] Full suite green: plugin build, vitest (≥33), root pytest, typecheck, skill lint
- [x] Woodpecker pipeline green on push
