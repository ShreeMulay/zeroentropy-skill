# Change: v1.1.3 Robust Transient Detection, Schema Test Coverage & Regex Scoping

## Status
completed

## Description
Land the high-leverage review cluster: replace fragile message-regex transient detection with SDK error-class detection (H1), add real-Zod validation tests for the `.max()` input limits (H2), and scope the status message-regex fallback to prevent false extraction (M-a).

## Motivation
- **H1 (bug, P1):** `TRANSIENT_MESSAGE_PATTERN` retries any error whose message contains "network"/"timed out"/etc. Empirically, `new Error("Configuration timed out waiting for user input")` is wrongly retried 5×. The SDK exports `APIConnectionError`/`APIConnectionTimeoutError` (status=undefined) which are the real transient errors — detect via `instanceof`.
- **H2 (task, P1):** Tests mock `@opencode-ai/plugin` with a no-op chain, so Zod `.min/.max/.enum/.default` never execute. The `.max(128/100/500_000)` guards have zero coverage.
- **M-a (bug, P2):** `getErrorStatus` regex `\b(4\d{2}|5\d{2})\b` can mis-extract a status from arbitrary numbers in an error message (e.g. "404 results found").

## Impact
- `plugin/src/index.ts` — import `APIConnectionError` from `zeroentropy`; `isTransientNetworkError` checks `instanceof APIConnectionError` first, then `.code` set, then a tightened message pattern (drop bare `network`/`timed? ?out`, keep explicit codes + "socket hang up"/"fetch failed"); scope `getErrorStatus` message fallback to a leading 3-digit code.
- `plugin/tests/index.test.ts` — add `APIConnectionError`-based retry test; add no-status non-network fail-fast test (the "timed out waiting for user input" case).
- `plugin/tests/schema.test.ts` (new) — real-Zod tests asserting `.max()` limits reject oversized input and `.min(1)` rejects empty.
- `CHANGELOG.md` — v1.1.3 entry.
